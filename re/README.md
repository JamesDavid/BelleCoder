# re/ — BLE reverse-engineering artifacts (SPEC §11)

The doll's BLE protocol was recovered by static analysis of the official Hasbro *Dance Code*
Android app (`com.hasbro.dancecode` v1.0.1). **Results: [`../docs/BLE_PROTOCOL.md`](../docs/BLE_PROTOCOL.md)** — that
doc is what fills the PAL.

## Layout
- `decompiled/` — the BLE-relevant C# recovered from `Assembly-CSharp.dll` (checked in; this is
  the source of truth for the opcode table). `BLEToyAPI.cs` and `AppToToyOpCodes.cs` are the
  key files; `startechplus_ble/` is the native-plugin bridge (shows the write type).
- `dancecode.apk`, `apk_x/`, `tools/` — regenerated by the scripts; **git-ignored** (the APK is
  ~88 MB and is not ours to redistribute).

## Reproduce from scratch
```bash
scripts/re/get_apk.sh        # download + verify size/sha/signing-cert
scripts/re/decompile.sh      # fetch dnSpy console, extract DLLs, decompile -> re/decompiled/
scripts/re/grep_ble.sh       # extract UUIDs / opcodes / write-type
python scripts/re/extract_assets.py   # 10 preset dances + phrase-duration table
```

## Extracted game data (checked in, outside re/)
- `assets/presets/dances/<name>_{dance,steps}.json` — the 10 Connect-dance choreographies
  (pure `{arm,l,r}` motion data; see `docs/BLE_PROTOCOL.md` for the format).
- `data/phrase_durations.csv` — 412 phrase indices → playback seconds, for `durationOf()`.

## Why this was easy
The app is Unity + **Mono** (not IL2CPP), so `Assembly-CSharp.dll` is ordinary .NET CIL and
decompiles to clean C#. No native reversing, no deobfuscation. The static decompile yielded the
full opcode table, both characteristic UUIDs, the adv-name filter (`DanceCD`), and the write
type — roughly 100% of the PAL, before any live GATT probing.

## Still needs the real doll (SPEC §11.4)
